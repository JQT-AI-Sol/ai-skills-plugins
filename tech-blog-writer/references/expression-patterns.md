# 表現パターン集

## 技術比喩パターン

専門用語を説明するときに使える比喩の例。読者層に合わせて使い分ける。

| 専門用語 | 比喩 | 使用例 |
|----------|------|--------|
| API | 窓口・受付カウンター | 「APIは、アプリ同士がやり取りするための窓口です」 |
| Docker | 持ち運べる作業環境 | 「Dockerは"自分のPC環境をそのまま箱詰め"できるツールです」 |
| Git | セーブポイント | 「Gitはコードのセーブポイント。いつでも前の状態に戻せます」 |
| CI/CD | 自動検品ライン | 「コードを書いたら自動でテスト・デプロイまで流れる検品ラインです」 |
| LoRA | 付箋メモ | 「モデル本体は書き換えず、付箋メモ（LoRA）で追加知識を貼り付けるイメージです」 |
| WebSocket | 常時接続の電話回線 | 「HTTPが毎回電話をかけ直すのに対し、WebSocketは回線つなぎっぱなしです」 |
| キャッシュ | メモ帳 | 「一度調べた結果をメモ帳に書いておいて、次回はメモを見るだけで済ませる仕組みです」 |
| コンテナ | 引っ越し用の段ボール | 「アプリと必要なものを段ボールに詰めて、どこでも同じように動かせます」 |
| ミドルウェア | 関所・チェックポイント | 「リクエストが本体に届く前に通る関所。認証やログ記録をここで処理します」 |
| 型システム | 注文書のフォーマット | 「"名前は文字列、個数は数値"と決めておく注文書。書き方を間違えるとエラーで教えてくれます」 |

**比喩の使い方のコツ:**
- 初心者向け記事では積極的に使う
- 経験者向けでは初出の概念のみに使う
- 「〜のようなものです」「イメージとしては〜」で自然に挿入

## 導入テンプレート

記事の書き出し。3パターンから記事タイプに合わせて選ぶ。

### パターンA: 課題提示型（チュートリアル・問題解決向き）

```markdown
{技術課題や困りごと}に直面したことはありませんか？

今回は{やったこと}の手順をまとめました。
{この記事で分かること}を解説します。
```

例:
> vLLMの推論が遅くて困っていませんか？
>
> 今回はバッチサイズとGPUメモリの最適化で推論速度を3倍にした手順をまとめました。
> 設定のポイントと実測値を解説します。

### パターンB: きっかけ型（体験記向き）

```markdown
{きっかけ・動機}がありました。
{何をやったか}を試してみたので、その過程を共有します。
```

例:
> Cloudflare Workersが気になっていたので、個人ブログを移行してみました。
> Vercelとの比較も含めて、移行の全過程を共有します。

### パターンC: 結論先出し型（比較レビュー・ディープダイブ向き）

```markdown
結論から言うと、{結論}でした。

{背景}という状況で{何を調べたか/試したか}を検証した結果です。
以下、詳しく見ていきます。
```

例:
> 結論から言うと、個人開発ならDocker Composeで十分でした。
>
> 「Kubernetesに移行すべきか？」を1ヶ月検証した結果です。
> 以下、判断基準と検証内容を詳しく見ていきます。

## 締めテンプレート

記事の締め方。3パターンから選ぶ。

### パターンA: まとめ+次のアクション型

```markdown
## まとめ

今回は{やったこと}を紹介しました。

ポイントをまとめると:
- {ポイント1}
- {ポイント2}
- {ポイント3}

{読者への次のアクション提案}。
```

### パターンB: 学び+展望型

```markdown
## 振り返り

{やってみて分かったこと}。

特に{一番の学び}は今後も使える知見だと思います。
次は{次に試したいこと}にも挑戦してみる予定です。
```

### パターンC: シンプル型（短い記事向き）

```markdown
## おわりに

{一言まとめ}。
同じ問題で困っている方の参考になれば幸いです。

リポジトリ/コード: {リンク}
```

**締めで避けること:**
- 「いかがでしたでしょうか」（AI臭い定番フレーズ）
- 「少しでもお役に立てれば幸いです」（月並み）
- 内容と関係ない一般論でふわっと終わる

## トラブルシューティングの書き方

エラーや躓きを記事に含めるときの表現パターン。

### 問題発生

```markdown
ここで問題が発生しました。

{エラーメッセージ or 症状}
```

- 「順調だったんですが、ここでハマりました」
- 「この設定を変えた途端、動かなくなりました」

### 原因の特定

```markdown
調べてみると、原因は{原因}でした。

{どうやって原因を特定したか}
```

- 「エラーメッセージでググると、同じ問題のIssueが見つかりました」
- 「ログをよく見ると、{具体的な箇所}が怪しいことに気づきました」

### 解決

```markdown
解決策は{解決策}です。

{修正前のコード/設定}
↓
{修正後のコード/設定}
```

- 「{設定値}を変更したら解決しました」
- 「公式ドキュメントの{セクション}に答えがありました」
