# Doc Condenser

AI生成Markdownドキュメントを簡潔・自然な文章に整えるレビュースキル。

## ワークフロー

### Step 1: 対象ファイルの特定

ユーザーからレビュー対象のファイルパスを受け取る。
複数ファイルの場合はファイル一覧を確認し、処理順を決める。

### Step 2: 全文読み取りと分析

ファイルを読み、以下の5カテゴリで問題を検出する:

**C1: 重複の統合**
- 同じ意味のセクション・段落が複数箇所にある
- 「背景」と「課題」、「目的」と「ゴール」等の実質同一セクション
- 導入と結論で同じ文の繰り返し

**C2: 冗長表現の圧縮**
- 回りくどい言い回し（「〜することが可能です」→「〜できる」）
- 不要な前置き・接続詞
- 過剰な修飾語（「非常に」「大幅に」「包括的な」等）
- [references/ai-expression-patterns.md](references/ai-expression-patterns.md) のパターンを参照

**C3: AI臭さの排除**
- バズワードの乱用（具体性のない「革新的」「シームレス」等）
- 過剰な丁寧表現
- 形容詞の羅列
- 箇条書き病（全てを箇条書きにする傾向）
- [references/ai-expression-patterns.md](references/ai-expression-patterns.md) のパターンを参照

**C4: 構造の最適化**
- 不要なセクションの削除提案（中身のない「はじめに」「まとめ」等）
- 細かすぎる見出し階層の統合
- 情報のグルーピング改善

**C5: 情報密度の向上**
- 同じことを言い換えて繰り返している箇所
- 具体性のない抽象的な記述（削除 or 具体化を提案）
- 文章で書くべき短い箇条書きの統合

### Step 3: レビュー結果の提示

以下のフォーマットで結果を提示する:

```
## レビュー結果: [ファイル名]

**削減見込み**: 約XX% (現在 約N行 → 推定 約M行)

### 指摘一覧

#### 1. [カテゴリ] 行XX-YY: [問題の要約]

**現状:**
> （該当箇所の引用）

**修正案:**
> （修正後のテキスト）

**理由:** （なぜこの修正が必要か、1文で）

---

#### 2. ...
```

指摘は影響度の大きい順（削減行数が多い順）に並べる。
軽微な表現修正は「その他の表現修正」としてまとめて提示してよい。

### Step 4: 承認と適用

ユーザーに以下を確認する:
- 「全て適用」「選択して適用（番号指定）」「修正案を調整」のいずれかを選択

承認された修正のみEditツールで適用する。
適用後、修正箇所数と削減行数をサマリーとして報告する。

## 注意事項

- 技術的に正確な記述は変更しない（用語の置き換えで意味が変わる場合は元のまま）
- コードブロック内は対象外
- 表やリストの構造的な意味がある箇所は保持する
- ドキュメントの対象読者（エンジニア / 非エンジニア / 経営層）を意識し、適切なトーンを維持する
- 「です・ます」調 / 「だ・である」調の統一は元の文体に合わせる
