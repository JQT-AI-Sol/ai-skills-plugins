# Tech Blog Writer

技術ブログの執筆を支援する。ユーザーが実践した内容（スクショ・メモ・参照URL）を基に、対話的にMarkdown記事を生成する。

**核心ルール: この記事はAIが書いたものではない。ユーザー本人が体験し、本人の言葉で書いた記事である。**

## ファイル保存ルール

すべてのブログ関連ファイルは `~/Work/` 配下に統一する。

| 種別 | 保存先 | 用途 |
|------|--------|------|
| 完成記事 | `~/Work/Qiita/public/` | Qiita CLI投稿用（FrontMatter付きMarkdown） |
| 素材・メモ | `~/Work/blog-materials/{topic-name}/` | スクショ、参考メモ、下調べ |
| 下書き（未完成） | `~/Work/blog-drafts/` | 執筆途中の記事 |

- 記事が完成したら必ず `~/Work/Qiita/public/` に保存する
- `blog-drafts/` は一時的な置き場。完成したら `public/` へ移動する

---

## 全体フロー

```
Phase 1: Plan（構成設計）
    ↓
Phase 2: Write（記事生成）
    ↓
Phase 2.5: Visual（図解生成）
    ↓
Phase 3: Review（品質検品）←─┐
    ↓                        │
  指摘あり？ ──Yes──→ Phase 4: Fix（修正）
    │                        │
    No                       │
    ↓                        │
  再Review ──────────────────┘
    │
  全パス
    ↓
Phase 5: Publish（仕上げ・保存）
```

Review ⇔ Fix は **最大3周** まで回す。3周で残る指摘は軽微なものとしてユーザーに提示して判断を委ねる。

---

## Phase 1: Plan（構成設計）

### 1-1. 資料の収集

ユーザーに以下を確認:
- **フォルダパス**: スクショやメモを保存したフォルダ（任意）
- **参照URL**: 公式ドキュメント、関連記事など（任意）
- **トピック**: 何について書くか

```
例:
トピック: Claude CodeでMCPサーバーを作った話
フォルダ: ~/blog-materials/mcp-server/
URL: https://modelcontextprotocol.io/docs
```

### 1-2. 資料の読み込み

1. 指定フォルダ内のファイルを確認（画像、テキスト、メモ）
2. 参照URLの内容をWebFetchで取得
3. 資料の概要を整理してユーザーに提示

### 1-3. 構成の検討

AskUserQuestionで以下を確認しながら構成を決定:

- 記事の目的（チュートリアル？体験記？レビュー？）
- 想定読者（初心者？経験者？）
- 伝えたいポイント
- 躓いたところはあったか

構成案を提示し、ユーザーと調整。

**記事タイプ別ガイド:**

| タイプ | 構成 | 目安語数 | 必須セクション | 適したテーマ |
|--------|------|:--------:|----------------|--------------|
| チュートリアル型 | 導入→前提条件→手順→まとめ | 2,000-4,000字 | TL;DR、前提条件、完成コード | ツールの使い方、セットアップ手順 |
| 体験記型 | きっかけ→やったこと→躓き→学び | 1,500-3,000字 | きっかけ、Before/After | 新技術への挑戦、やってみた系 |
| 比較レビュー型 | 背景→比較軸→各ツール評価→結論 | 2,000-3,500字 | 比較表、結論（どれを選ぶべきか） | ツール比較、技術選定 |
| 問題解決型 | 課題→調査→解決策→結果 | 1,000-2,000字 | エラー内容、原因、解決策 | エラー対応、トラブルシュート |
| ディープダイブ型 | 概要→仕組み→実装詳細→トレードオフ | 3,000-5,000字 | 図解、トレードオフ | アーキテクチャ解説、内部実装 |

### 1-4. タイトルの作成

記事の構成が決まったら、タイトル候補を3つ提示する。

**タイトル作成ルール:**
- 30-50文字を目安（長すぎず短すぎず）
- 具体的な技術名・ツール名を含める
- 読者が得られるベネフィットを示す

**タイトルパターン:**

| パターン | 例 |
|----------|-----|
| How to型 | 「Claude CodeでMCPサーバーを構築する方法」 |
| 数字活用型 | 「vLLM推論を3倍速くした5つの工夫」 |
| 【】強調型 | 「【完全ガイド】LoRAファインチューニング入門」 |
| 体験型 | 「Next.js + Cloudflareで個人ブログを作り直した話」 |
| 問題解決型 | 「CUDAメモリ不足エラーを解決するまでの記録」 |
| 疑問型 | 「Docker ComposeとKubernetes、個人開発ならどっち？」 |

## Phase 2: Write（記事生成）

以下のガイドに従い記事を生成:

- [references/style-guide.md](references/style-guide.md) — 文体・表記・禁止表現・タグ選定・AI表現パターン
- [references/expression-patterns.md](references/expression-patterns.md) — 比喩・導入/締めテンプレート

**生成時の心構え:**

この記事の筆者はユーザー本人。AIはゴーストライター。
- ユーザーが実際に体験したこと「だけ」を書く
- 体験していないことは書かない。わからないことは「わからなかった」と書く
- ユーザーの口調・テンションに合わせる
- 「教えてあげる」ではなく「やってみたらこうだった」のスタンス

**生成時チェック:**
- カジュアルで読みやすい文体になっているか
- 専門用語は簡単な説明付きか
- スクショの挿入位置を `![説明](ファイル名)` で明示しているか
- コード例が品質基準を満たしているか（style-guide参照）
- 記事が目安語数を大幅に超えていないか（超える場合は分割を提案）

## Phase 2.5: Visual（図解生成）

記事のテキストが完成したら、図解が効果的な箇所を特定し、画像を生成する。
ユーザーのスクショがメイン素材。AI生成は**概念の可視化**（フロー図、構成図等）だけに使う。

### 2.5-1. 画像の棚卸し

まず記事内の画像状況を確認する:

1. ユーザーが素材フォルダに用意したスクショを確認（Phase 1-2で読み込み済み）
2. 記事内の `![` の挿入位置を確認
3. **スクショが足りない箇所**と**図解が欲しい箇所**を分けてリストアップ

| 画像の種類 | 対応方法 |
|-----------|---------|
| スクショ（画面キャプチャ） | ユーザーの素材フォルダから配置。足りなければpeekabooで追加撮影 |
| 図解（フロー図、構成図） | Mermaid記法で記述。Mermaidで表現しにくい場合のみAI画像生成 |
| コード実行結果 | コードブロックで十分。スクショ不要 |

### 2.5-2. Mermaid記法での図解（優先）

QiitaはMermaid記法をサポートしている。AI画像生成よりMermaidを**優先**する。

**Mermaidが適するケース:**

| 記事の内容 | Mermaid記法 |
|-----------|------------|
| ワークフロー・手順 | `graph LR` / `graph TD` |
| シーケンス（API呼び出し等） | `sequenceDiagram` |
| アーキテクチャ構成 | `graph TD` + subgraph |
| 状態遷移 | `stateDiagram-v2` |
| タイムライン | `timeline` |

```markdown
例: アーキテクチャ図
​```mermaid
graph TD
    A[ブラウザ] --> B[Next.js]
    B --> C[Supabase]
    B --> D[Lambda]
    D --> E[OpenAI API]
​```
```

### 2.5-3. AI画像生成（Mermaidで不十分な場合のみ）

Mermaidでは表現しにくい以下のケースでのみ、image-generatorスキルでAI画像を生成する:

- 比較マトリクス（2x2グリッド）
- Before/After比較図
- インフォグラフィック（数値データの視覚化）
- 概念図（抽象的な関係性）

**pdf-docのPhase 1.5に準拠した生成フロー:**

1. **プロンプト作成**: 記事内容から具体的なラベル・数値を抽出してプロンプトに含める
2. **生成**: image-generatorスキル（NanoBanana Pro 2）で生成
3. **検品**: Read toolで画像を確認。以下の3点チェック:
   - 文字が読めるか（文字化けは即リトライ）
   - 内容が記事と一致しているか
   - レイアウトが崩れていないか
4. **リトライ**: 不合格なら最大3回再生成。3回ダメならスキップ

**プロンプトの注意:**
- 白背景、シンプルなフラットデザイン（ブログ記事に馴染むように）
- `"1200x600px, white background, minimal flat design, clear sans-serif font"` をベースに
- 写真風・リアル風は禁止（AI臭くなる）

### 2.5-4. 画像の配置

生成・収集した画像を記事に配置する。

```markdown
![セットアップ画面](screenshots/01-setup.png)
```

**配置ルール:**
- 画像枚数の目安: 記事全体で **3-5枚**（多すぎると読みにくい）
- スクショは手順の直後に配置
- 図解はセクション冒頭（概要説明の直後）に配置
- 全セクションに画像を入れる必要はない

**Qiita画像アップロードの注意:**
- Qiita CLIでは画像アップロード不可
- 画像は記事公開時にQiita Web上で手動追加する
- Markdownには `![alt](ファイル名)` のプレースホルダーを残しておく

### 2.5-5. スキップ条件

以下の場合は図解生成をスキップする:
- 記事が短い（1,000字未満）
- 問題解決型の記事（コードとエラーメッセージで十分）
- ユーザーが素材フォルダに十分なスクショを用意済み

## Phase 3: Review（品質検品）

**記事をファイルに保存してから** レビューを実施する（下書き段階では `~/Work/blog-drafts/` に保存）。
保存したファイルをReadで読み直し、以下の6カテゴリで全文をチェックする。

詳細なチェック基準: [references/review-checklist.md](references/review-checklist.md)

### チェックカテゴリ

| ID | カテゴリ | 致命度 | 概要 |
|----|----------|:------:|------|
| R1 | 知ったかぶり検出 | **致命** | 資料にない情報を断言していないか |
| R2 | 上から目線検出 | **致命** | 読者を見下す表現がないか |
| R3 | AI臭さ検出 | **重大** | AI生成テキスト特有の冗長・不自然パターン |
| R4 | 冗長・重複 | 中 | 同じことの繰り返し、回りくどい表現 |
| R5 | 構造の最適化 | 中 | 不要セクション、過剰な見出し階層 |
| R6 | 情報密度 | 軽微 | 抽象的な記述、具体性の欠如 |

### レビュー結果の出力フォーマット

```
## Review Round N: [ファイル名]

**判定**: PASS / FAIL
**致命的指摘**: X件 | **重大指摘**: Y件 | **中**: Z件 | **軽微**: W件

### 指摘一覧

#### 1. [R1: 知ったかぶり] 行XX: [問題の要約]
**現状:** > （該当箇所の引用）
**問題:** （なぜダメか）
**修正案:** > （修正後のテキスト）

---
#### 2. ...
```

**PASSの条件**: 致命0件 かつ 重大0件

---

## Phase 4: Fix（修正）

Review結果に基づき修正を適用する。

### 修正の優先順

1. **致命（R1, R2）を最優先で修正** — 知ったかぶり・上から目線は1件でも残ったらFAIL
2. **重大（R3）を修正** — AI臭さは全箇所修正
3. **中・軽微（R4-R6）** — まとめて修正

### 修正ルール

- Editツールで直接修正する（ユーザーに修正案を見せて承認を待つのではなく、即修正）
- 修正後、Phase 3に戻って再Reviewする
- **修正で新たな問題を作らない**: 表現を変えたことで意味が変わっていないか確認

### ループ制御

- Review ⇔ Fix は **最大3周**
- 3周目でまだFAILなら、残りの指摘をユーザーに提示して判断を仰ぐ
- 各ラウンドの指摘件数を記録し、減少傾向でなければアプローチを変える

## Phase 5: Publish（仕上げ・保存）

Reviewが全PASSしたら、以下を順に実行する。

### 5-1. FrontMatterの生成

Qiita投稿用のYAML FrontMatterを記事冒頭に必ず含める。

```yaml
---
title: 記事タイトル
tags:
  - タグ1
  - タグ2
  - タグ3
  - タグ4
  - タグ5
private: true
updated_at: ''
id: null
organization_url_name: jqiit-co
slide: false
ignorePublish: false
---
```

**FrontMatterルール:**
- `private: true` で必ず下書き保存（公開はQiita Webで手動）
- タグは必ず5つ。選定基準は style-guide.md の「タグ選定ガイド」参照
- `organization_url_name: jqiit-co` を必ず設定

### 5-2. 関連記事リンクの追加

記事末尾に「## 関連記事」セクションを追加する。

**ルール:**
- 同じシリーズの記事がある場合は前後リンクを必ず双方向で設定
- 同じ技術スタック（Claude Code、ローカルAI等）の記事へのリンクを2-3本追加
- リンク形式: `- [記事タイトル](https://qiita.com/TMiyamoto/items/{id}) — 補足説明`
- `~/Work/Qiita/public/` 内のMarkdownファイルのFrontMatterからタイトルとIDを取得して使う

### 5-3. 完成記事の保存

`~/Work/blog-drafts/` から `~/Work/Qiita/public/` に移動する。

### 5-4. X投稿素材の更新

X Growth スキルの投稿素材ファイルを更新する。

**対象ファイル:** `~/.claude/skills/x-growth/references/my-profile.md`

**更新内容:**
1. 「Qiita記事一覧」テーブルに新しい記事を追加（記事タイトル、URL/ID、使える場面）
2. 「語れるテーマと具体エピソード」に新しいテーマがあれば追加
- 記事IDは `~/Work/Qiita/public/` 内のファイル名から取得

## 入力フォーマット

### 起動時の指示例

```
ブログを書きたい
トピック: AgentBrowserを試してみた
フォルダ: ~/blog-materials/agentbrowser/
参照URL:
- https://github.com/anthropics/agentbrowser
```

### フォルダ構成例（任意）

```
~/blog-materials/topic-name/
├── screenshots/
│   ├── 01-setup.png
│   └── 02-result.png
└── notes.md
```

### デスクトップスクショ（peekaboo）

ブログ素材でターミナル出力やIDE画面のスクショが必要な場合:

```bash
peekaboo image --path /tmp/blog-screenshot.png
```

Playwrightはブラウザ限定。peekabooはmacOS全体をカバー。

## 出力フォーマット

- Markdown形式
- H2/H3で構造化
- コードブロックでコマンド/コードを表示
- 重要キーワードは**太字**
- スクショ挿入位置を `![alt](path)` で明示

## 注意事項

- 資料にない情報は推測せずAskUserQuestionで確認
- 構成はユーザーと相談して決める（押し付けない）
- 専門用語は読者層に合わせて説明レベルを調整
