# スタイルガイド

## 基本方針

技術ブログをカジュアルで読みやすい文体で執筆する。
「同僚のエンジニアに話しかけるトーン」が基準。

## 文体ルール

### 語尾
- 「〜です・ます」調が基本
- 適度に「〜ですね」「〜ですよね」と読者に語りかける
- 硬すぎず、崩しすぎない

### リズム
- 文は短く切る
- 体言止めを効果的に使う
- 一文一意を心がける

### 表現
- 専門用語は初出時に簡単な説明を添える
- 比喩を使って分かりやすく（→ expression-patterns.md参照）
- 口語表現を適度に混ぜてOK

### 感情表現の指針

技術者としての素直な反応を適度に入れる。ただしblog-writerほどは崩さない。

**使ってOKな表現:**
- 驚き: 「これは予想外でした」「思ったより簡単でした」「地味にハマりました」
- 感想: 「個人的にはかなり好みです」「開発体験がだいぶ変わりました」
- 苦戦: 「ここで30分溶かしました」「ドキュメントが見つからず苦労しました」
- 成功: 「無事動きました」「期待通りの結果です」

**避けるべき表現:**
- 過度な興奮: 「最高すぎる！！」「神ツール！」
- 自虐の多用: 「こんな簡単なことも分からない自分...」
- 無意味な感嘆符の連打: 「すごい！！！」

## 表記ルール

### 見出し
- H2、H3を明確に使い分ける
- 数字を使うと読みやすい（「3つのポイント」など）

### 強調
- 重要なキーワードは**太字**
- 専門用語やファイル名は`インラインコード`
- コマンドやコードは```コードブロック```

### 画像
- `![説明文](ファイルパス)` 形式
- 説明文は画像の内容を簡潔に

## コード例の品質基準

記事内のコード例は以下を守る:

1. **実行可能であること** — コピペで動くコードを載せる。省略する場合は `// ...省略...` と明示
2. **バージョンを明記** — 「Python 3.11 / Node 20 で動作確認」など、記事冒頭かコード付近に記載
3. **出力結果を表示** — コマンドやスクリプトの実行結果もコードブロックで載せる
4. **エラーハンドリングは文脈次第** — チュートリアルでは省略OK、本番向けコードでは含める
5. **言語タグを付ける** — ` ```python ` ` ```bash ` など、シンタックスハイライト用

```bash
# 良い例
$ npm create next-app@latest my-app -- --typescript
# 出力:
# ✔ Would you like to use ESLint? … Yes
# ✔ Would you like to use Tailwind CSS? … Yes
```

## タグ選定ガイド

Qiita記事のタグは必ず5つ設定する。フォロワー数の多いタグを優先的に使うことで検索流入を最大化する。

### 必須タグ（該当すれば必ず含める）

| タグ | フォロワー数 | 対象 |
|------|:-----------:|------|
| `Python` | 400K+ | Pythonコードを含む記事 |
| `生成AI` | 高 | AI/LLM/Claude/GPT関連すべて |
| `AI` | 高 | AI関連すべて（生成AIと併用OK） |

### 推奨タグ（テーマに合わせて選択）

| タグ | 対象 |
|------|------|
| `ClaudeCode` | Claude Code関連 |
| `AIエージェント` | エージェント・マルチエージェント |
| `機械学習` | ML/ファインチューニング |
| `ファインチューニング` | SFT/LoRA/DPO |
| `自動化` | ワークフロー自動化 |
| `Playwright` | ブラウザ自動化 |
| `Next.js` | Next.js/React |

### 避けるべきタグ

- フォロワーが極端に少ないニッチタグ（`NanoBananaPro`, `sft`, `Unsloth`等）
- 一般的すぎて差別化できないタグ（`プログラミング`, `勉強`等）
- 固有名詞の略称（`SKILLS`, `松尾研`等）→ より汎用的なタグに置き換える

### タグ選定の手順

1. 記事の主要技術を特定
2. 必須タグから該当するものを選ぶ（1-3個）
3. 推奨タグから残り枠を埋める（2-4個）
4. 合計5つになるよう調整

## 記事の長さ制御

目安語数を超えそうな場合の対処:

- **200行超**: 内容を精査。冗長な説明がないか確認
- **300行超**: セクションの分割・折りたたみ（`<details>`タグ）を検討
- **400行超**: 前編/後編に分割を提案

テスト結果の比較など、同じ構造の繰り返しは2例目以降を`<details>`で折りたたむ。

## 避けるべき表現

### AI臭い表現（禁止）

以下はAI生成テキストに頻出するパターン。記事に残っていたら必ず書き換える。

**禁止フレーズ → 置換例:**

| 禁止 | 置換 |
|------|------|
| 〜について深掘りしていきましょう | 〜を見ていきます / 〜を詳しく解説します |
| 重要なのは〜ということです | （削除して直接述べる） |
| 〜と言えるでしょう | 〜です |
| まず初めに | まず |
| 〜することが可能です | 〜できます |
| 〜を活用する | 〜を使う |
| 〜を実現する | 〜する / 〜を作る |
| さまざまな〜 | （具体的に列挙する） |
| 〜において | 〜で / 〜では |
| 〜という観点から | （削除するか具体的に） |
| それでは早速〜 | （削除して本題に入る） |
| いかがでしたでしょうか | （削除。締めはexpression-patterns.md参照） |

**禁止パターン:**
- 「〜のメリットとデメリット」で始まる見出し → 具体的な内容を見出しにする
- 箇条書きが全て「〜すること」で終わる → 体言止めか動詞で変化をつける
- 段落の冒頭が全て「また、」「さらに、」「加えて、」 → 接続詞に頼らず文脈でつなぐ
- 「〜の世界」「〜の領域」（landscape の日本語訳） → 削除するか具体的に
- 結論が「〜は非常に強力なツールです」 → 具体的な評価を述べる

### その他の避けるべき表現
- 過度に堅い敬語
- 曖昧な表現（「〜かもしれません」の多用）
- 専門用語の説明なしでの使用
- 「簡単に」「誰でも」などの安易な断言（読者のレベルによる）

## 冗長表現 → 簡潔表現

記事全体を通して、冗長な表現は簡潔に言い換える。

| 冗長 | 簡潔 |
|------|------|
| 〜を行う | 〜する |
| 〜を実施する | 〜する |
| 〜に関して | 〜について / 〜の |
| 〜という形で | 〜で |
| 〜のような場合 | 〜なら |
| 基本的には〜 | 〜（削除可） |
| 現時点では〜 | 今は〜 |
| 〜を確認することができます | 〜を確認できます |
| 〜する必要があります | 〜が必要です / 〜します |
| そのため〜 | なので〜 / だから〜 |
