# Review Checklist（品質検品基準）

記事生成後のレビューで使用する。各カテゴリの指摘を検出し、修正する。

## R1: 知ったかぶり検出（致命）

**AIが最もやりがちな致命的問題。** ユーザーが体験していないことをさも体験したかのように書くこと。

### 検出パターン

| パターン | 例 | なぜダメか |
|----------|-----|-----------|
| 資料にない具体的数値 | 「レスポンスが50ms改善」 | ユーザーが計測していないなら嘘 |
| 資料にない比較 | 「〇〇より高速」 | 比較していないなら知ったかぶり |
| 体験していない感想 | 「開発体験が劇的に向上した」 | 素材に「良かった」とあっても「劇的に」は盛りすぎ |
| 一般論を体験談風に書く | 「実際に使ってみると〜だと感じました」 | その「実際に」は本当にユーザーがやったか？ |
| 裏付けのない断言 | 「本番環境でも問題なく動作します」 | 本番で試してなければ嘘 |
| ドキュメントの受け売り | 「〇〇は△△を実現する機能です」 | 公式の説明をそのまま体験談に混ぜると知ったかぶりに見える |

### 修正方針

- **資料にある事実だけを書く**。資料に「速くなった気がする」とあるなら「速くなった気がする」と書く。「3倍速くなった」に盛らない
- 一般的な情報は「公式ドキュメントによると〜」「READMEには〜と書いてある」と出典を明示する
- わからないことは「この辺はまだ検証できていない」と正直に書く
- **疑わしければAskUserQuestionで確認する**

## R2: 上から目線検出（致命）

**読者に「お前に言われたくない」と思わせる表現。** AIは「教えてあげる」モードになりやすい。

### 検出パターン

| パターン | 例 | 修正例 |
|----------|-----|--------|
| 命令・指示口調 | 「〜してください」「〜すべきです」 | 「〜するとよさそうです」「自分は〜しました」 |
| 断定的な評価 | 「これは必須です」「〜は使うべきではありません」 | 「自分の場合は〜が必要でした」 |
| 読者を初心者扱い | 「ご存知の通り〜」「言うまでもなく〜」 | 削除。知ってるなら書く必要がない |
| 上位者の視点 | 「チームに導入することをお勧めします」 | 「自分のチームでは〜してみました」 |
| 一般化した教訓 | 「〇〇開発では△△が重要です」 | 「今回は△△がポイントだった」 |
| 網羅的な解説 | 「〇〇のメリット・デメリットを解説します」 | 「使ってみて良かった点・困った点」 |
| 予防的な注意喚起 | 「注意: 〜に気をつけてください」 | 「自分はここでハマりました」 |

### 修正方針

- **主語を「自分」「今回」にする**。「〜すべき」→「自分は〜した」「今回は〜にした」
- 教える側ではなく**共有する側**のスタンス。「やってみたらこうだった」
- 読者と同じ目線。「自分もよくわかってなかったんですが」はOK
- 失敗や迷いも正直に書く。完璧な人間は嘘くさい

## R3: AI臭さ検出（重大）

AI生成テキストに共通する不自然さ。人間はこういう書き方をしない。

### 3-1. 冗長表現

| AI表現 | 人間の表現 |
|--------|-----------|
| 〜することが可能です | 〜できる |
| 〜を実現することができます | 〜できる |
| 〜を行うことが重要です | 〜が大事 |
| 〜について説明いたします | 〜を説明する |
| 〜という点が挙げられます | 〜がある |
| 〜と言えるでしょう | 〜だ |
| 〜であると考えられます | 〜だろう |
| 〜に関して述べますと | 〜は |
| 〜を目的として | 〜のため |
| 〜のプロセスを経て | 〜して |
| 〜を前提とした場合 | 〜なら |
| 〜という観点から見ると | 〜の面では |
| 〜を活用する | 〜を使う |
| 〜を実現する | 〜する / 〜を作る |
| さまざまな〜 | （具体的に列挙する） |
| 〜において | 〜で |

### 3-2. 不要な前置き・接続

| AI表現 | 修正 |
|--------|------|
| まず最初に | まず |
| それでは次に | 次に |
| ここで重要なのは | （直接述べる） |
| 以下に詳しく説明します | （直接説明に入る） |
| 上記を踏まえた上で | これを踏まえ |
| 結論から申し上げますと | 結論として |
| それでは早速〜 | （削除して本題に入る） |

### 3-3. 無意味な強調・バズワード

| AI表現 | 修正 |
|--------|------|
| 非常に重要な | 重要な（本当に強調が必要な場合のみ） |
| 大幅に向上させる | 向上させる |
| 包括的なアプローチ | （削除。具体性がない） |
| シームレスに統合 | 統合 |
| 効果的に活用する | 使う |
| 革新的な | （具体的に何が新しいか書く） |
| 堅牢な | （具体的な強度を書く） |
| 最適な〜を実現 | 〜を実現 |

### 3-4. AI特有の構造パターン

- **箇条書き病**: 全てを箇条書きにする → 文章で流せるものは文章にする
- **エコー**: 導入で述べた内容を結論でほぼ同文で繰り返す → 結論では新しい視点を加えるか、短くまとめる
- **過剰な構造化**: 2-3文で済む内容にH2/H3の見出し → 見出しを減らして文章にまとめる
- **同義セクション**: 「背景」と「課題」で同じこと → 1つに統合
- **空のまとめ**: 本文の再掲でしかない「まとめ」 → 削除するか、次のアクションだけ書く

### 3-5. 禁止フレーズ

以下は検出したら即修正:

| 禁止 | 理由 |
|------|------|
| いかがでしたでしょうか | AI記事の代名詞 |
| 〜について深掘りしていきましょう | AI臭い誘導 |
| 少しでもお役に立てれば幸いです | 月並み |
| 〜は非常に強力なツールです | 具体性ゼロ |
| 〜のメリットとデメリット（見出し） | AI記事テンプレ |
| 〜の世界 / 〜の領域 | "landscape"の直訳 |

## R4: 冗長・重複（中）

### 検出パターン

- 同じ意味のセクション・段落が複数箇所にある
- 導入と結論で同じ文の繰り返し
- 前のセクションの要約で次のセクションが始まる
- 同じことを言い換えて繰り返している

### 修正方針

- 重複箇所は一方を削除し、もう一方に情報を集約する
- セクション間の橋渡しは1文で十分

## R5: 構造の最適化（中）

### 検出パターン

- 中身のない「はじめに」「まとめ」
- 細かすぎる見出し階層（H4以深は基本不要）
- 1-2文しかないセクション
- 情報のグルーピングが不適切

### 修正方針

- 不要セクションは削除
- 薄いセクションは隣接セクションに統合
- H4以深はH3に昇格させるか、箇条書きにする

## R6: 情報密度（軽微）

### 検出パターン

- 具体性のない抽象的な記述（「様々な利点がある」→何が？）
- 文章にすべき短い箇条書きの羅列
- 修飾語を取っても意味が変わらない文

### 修正方針

- 抽象 → 具体化するか削除
- 修飾語を削ってみて意味が同じなら削る
