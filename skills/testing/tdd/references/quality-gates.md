# Quality Gates: エージェント間品質保証システム

> **3つのエージェントロール（Planner / Generator / Evaluator）が各工程で品質を担保する。**
> Evaluator は各ゲートでスコアリングを行い、合否を判定する。

---

## エージェントロール定義

### Planner（計画者）

**責務**: 何をテストすべきか、どうレビューすべきかを設計する。

- テスト対象の分析と優先度付け
- テスト種別の判定とシナリオ設計
- レビュー方針の策定（どのスキルを呼ぶか）
- 品質基準の事前設定

**担当ゲート**: Gate 1（コンテキスト分析）、Gate 2（テスト種別判定）、Gate 3（コードレビュー計画）

### Generator（実行者）

**責務**: Planner の計画に基づき、コード・テスト・ドキュメントを実装する。

- テストコードの作成（RED）
- 本番コードの実装（GREEN）
- リファクタリング（REFACTOR）
- ドキュメント同期

**担当ゲート**: Gate 4（RED）、Gate 5（GREEN）、Gate 6（REFACTOR）、Gate 9（ドキュメント同期）

### Evaluator（評価者）

**責務**: 各ゲートの成果物をスコアリングし、合否を判定する。

- 評価基準に基づくスコアリング（0-3点 × 項目数）
- 合否判定（PASS / CONDITIONAL / FAIL）
- 差し戻し時の具体的指摘（どの項目が不足か）
- 最終品質レポートの出力

**全ゲートで Evaluator が評価を実行する。**

---

## スコアリング体系

### 採点基準

| 点数 | レベル | 意味 |
|------|--------|------|
| **3** | Excellent | 完全に満たしている。改善の余地なし |
| **2** | Good | 概ね満たしている。軽微な改善点あり |
| **1** | Partial | 部分的に満たしている。重要な不足あり |
| **0** | Missing | 満たしていない、または未実施 |

### 合否判定

```
スコア = (獲得点数合計 / 満点) × 100
```

| スコア | 判定 | アクション |
|--------|------|-----------|
| **80-100** | PASS | 次の工程へ進む |
| **60-79** | CONDITIONAL | 指摘箇所のみ修正 → Evaluator が再評価 |
| **0-59** | FAIL | 工程やり直し（Planner に差し戻し） |

### Critical 項目ルール

各ゲートに `[CRITICAL]` マーク付き項目がある場合:
- **Critical 項目が 0点 → スコアに関係なく FAIL**
- Critical 項目は「これが欠けると後工程が破綻する」品質の最低ライン

### フィードバックループ

```
Planner（計画）
    ↓
Generator（実行）
    ↓
Evaluator（評価）
    ├── PASS → 次のゲートへ
    ├── CONDITIONAL → Generator（部分修正 → 再評価）
    └── FAIL → Planner（計画見直し → Generator → 再評価）
```

CONDITIONAL での再評価は **最大2回**。2回修正しても CONDITIONAL 以下なら FAIL に格上げ。

## Gate 1: コンテキスト分析（Section 1）

**担当**: Planner → Evaluator
**目的**: 変更内容を正確に把握し、テスト戦略の基盤を作る

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | `[CRITICAL]` 変更ファイルの網羅的特定 | 3 | git diff / status で全変更ファイルをリストアップしているか |
| 2 | モード判定の正確性 | 3 | 差分テスト / カバレッジ監査の判定が妥当か |
| 3 | 変更の影響範囲分析 | 3 | 変更が他のモジュールに与える影響を把握しているか |
| 4 | テスト対象の明確化 | 3 | 何をテストすべきかが具体的に特定されているか |

**満点**: 12点 → **合格ライン**: 10点（80%）

---

## Gate 2: テスト種別判定（Section 2）

**担当**: Planner → Evaluator
**目的**: 適切なテスト種別と配置先を決定する

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | `[CRITICAL]` テスト種別の正確性 | 3 | パスパターンに基づく判定が正しいか（単体/API/E2E） |
| 2 | POM存在確認（E2E時） | 3 | E2E判定時に POM の有無を確認し、不足分の対応方針があるか |
| 3 | 既存テストとの整合性 | 3 | 既存テストの更新 vs 新規作成の判断が適切か |
| 4 | テスト配置先の正確性 | 3 | `tests/unit/` `tests/api/` `tests/e2e/` への振り分けが正しいか |

**満点**: 12点 → **合格ライン**: 10点（80%）

---

## Gate 3: コードレビュー（Section 2.5）

**担当**: Planner（レビュー方針策定）→ Generator（レビュースキル実行）→ Evaluator
**目的**: テスト前にソースコードの品質と構造を担保する

### 3a: 構造チェック

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | `[CRITICAL]` ファイル行数チェック | 3 | 300行超のファイルがない、または `/decompose` で分割済み |
| 2 | 責務分離 | 3 | 1ファイルに複数の責務が混在していないか |
| 3 | ビジネスロジックの配置 | 3 | コンポーネント内にロジックが直書きされていないか |

### 3b: ソースコードレビュー

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 4 | `[CRITICAL]` Critical指摘ゼロ | 3 | レビュースキルの Critical 指摘が全て解消されているか |
| 5 | レビュースキルの適切な選択 | 3 | 変更パスに応じた正しいスキルが呼ばれているか |
| 6 | セキュリティ観点 | 3 | XSS / SQLi / CSRF 等の脆弱性がないか |

### 3c: DBレビュー（該当時のみ）

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 7 | RLSポリシー確認 | 3 | tenant_id による行レベルセキュリティが適切か |
| 8 | インデックス確認 | 3 | FK列・検索条件列にインデックスがあるか |

**満点**: 24点（DB変更なし: 18点）→ **合格ライン**: 80%
**スキップ条件適用時**: Gate 3 自体をスキップ（テスト/ドキュメント/設定のみの変更）

## Gate 4: RED フェーズ（Section 3）

**担当**: Generator → Evaluator
**目的**: 正しい失敗テストが作成されていることを検証する

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | `[CRITICAL]` テストが失敗している | 3 | テスト実行結果が FAIL であること（実際のログで確認） |
| 2 | `[CRITICAL]` 失敗理由がアサーション失敗 | 3 | コンパイルエラー/importエラーではなく、アサーション失敗で落ちている |
| 3 | AAAパターン準拠 | 3 | Arrange / Act / Assert の構造が明確か |
| 4 | テスト名の記述性 | 3 | テスト名が「何の振る舞いを検証するか」を明示しているか |
| 5 | 正常系テストの存在 | 3 | 期待どおりの入力→出力のテストがあるか |
| 6 | 境界値・異常系テスト | 3 | 空値、不正入力、エッジケースのテストがあるか |
| 7 | 1テスト1アサーション | 3 | 原則として1テストに1つのアサーションか |
| 8 | テストの独立性 | 3 | テスト間に実行順序依存がないか |

**満点**: 24点 → **合格ライン**: 20点（80%）

---

## Gate 5: GREEN フェーズ（Section 4）

**担当**: Generator → Evaluator
**目的**: 最小限の実装でテストがパスしていることを検証する

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | `[CRITICAL]` 新テストがパス | 3 | 新しく追加したテストが全てパスしている（実際のログで確認） |
| 2 | `[CRITICAL]` 既存テスト非破壊 | 3 | 既存テストが壊れていない |
| 3 | `[CRITICAL]` 実行ログの提示 | 3 | 「たぶん通る」ではなく、実際のテスト実行ログが提示されている |
| 4 | YAGNI準拠 | 3 | テストを通すために必要最小限のコードだけが追加されているか |
| 5 | テストバイパス不在 | 3 | テスト自体を書き換えてパスさせていないか |

**満点**: 15点 → **合格ライン**: 12点（80%）

---

## Gate 6: REFACTOR フェーズ（Section 5）

**担当**: Generator → Evaluator
**目的**: コード品質が改善され、テストが壊れていないことを検証する

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | `[CRITICAL]` 全テストパス維持 | 3 | リファクタ後も全テストがパスしている（実際のログで確認） |
| 2 | 重複コード除去 | 3 | 明らかな重複が残っていないか |
| 3 | 命名の適切性 | 3 | 変数・関数名が意図を正確に表現しているか |
| 4 | 関数の責務分離 | 3 | 1関数1責任になっているか |
| 5 | テストコードの品質 | 3 | 共通セットアップの抽出、ヘルパーの適切な利用ができているか |
| 6 | スコープ逸脱なし | 3 | 新機能追加やパフォーマンス最適化が混入していないか |

**満点**: 18点 → **合格ライン**: 15点（80%）

---

## Gate 7: 目視レビュー（Section 5.5、E2E時のみ）

**担当**: Generator（動画撮影）→ Evaluator（チェックリスト評価）
**目的**: UIの見た目と操作性を人間の目で確認する

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | レイアウト崩れなし | 3 | 要素の重なり、はみ出し、意図しない余白がないか |
| 2 | レスポンシブ対応 | 3 | モバイル / タブレット / デスクトップで崩れないか |
| 3 | インタラクション正常 | 3 | クリック、入力、遷移が期待どおり動作するか |
| 4 | ローディング・エラー表示 | 3 | 非同期処理中の表示、エラー時のフィードバックが適切か |
| 5 | アクセシビリティ基本 | 3 | コントラスト比、フォーカス順序、alt属性が適切か |

**満点**: 15点 → **合格ライン**: 12点（80%）
**スキップ条件**: 単体/APIテストのみ、またはユーザーが「動画不要」と指示

---

## Gate 8: 最終検証（Section 8）

**担当**: Evaluator（最終判定）
**目的**: TDDサイクル全体の品質を総合的に検証する

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | `[CRITICAL]` 全テスト実行ログ | 3 | `npx vitest run` / `npm run test:e2e` の実行ログが提示されている |
| 2 | `[CRITICAL]` 全テストパス | 3 | 出力に FAIL がゼロ |
| 3 | テストファースト遵守 | 3 | 全ての本番コードに先行するテストが存在する |
| 4 | 禁止表現なし | 3 | 「おそらく通る」「問題ないはず」等の曖昧表現が使われていない |

**満点**: 12点 → **合格ライン**: 10点（80%）

---

## Gate 9: ドキュメント同期（Section 9）

**担当**: Generator（ドキュメント更新）→ Evaluator
**目的**: コード変更に対応するドキュメントが最新であることを検証する

| # | 評価項目 | 配点 | 評価基準 |
|---|---------|------|---------|
| 1 | 同期対象の網羅性 | 3 | 変更に影響するドキュメントが全て特定されているか |
| 2 | API定義書の最新性 | 3 | エンドポイント・リクエスト・レスポンスが実装と一致しているか |
| 3 | DB定義書の最新性 | 3 | テーブル・カラム・RLSが実装と一致しているか |
| 4 | 画面遷移図の最新性 | 3 | 画面ノード・遷移パスが実装と一致しているか |
| 5 | テスト仕様書の最新性 | 3 | テストケース・実装一覧が実装と一致しているか |

**満点**: 15点（該当項目のみで計算）→ **合格ライン**: 80%
**スキップ条件**: リファクタのみ / テストヘルパーのみ / CSS調整のみ / 依存更新のみ

## 品質レポートフォーマット

各ゲート通過時、Evaluator は以下のフォーマットでレポートを出力する。

```
## Quality Gate Report: Gate X - [ゲート名]

| # | 評価項目 | 点数 | コメント |
|---|---------|------|---------|
| 1 | [項目名] | X/3 | [具体的な評価理由] |
| 2 | ... | ... | ... |

**スコア**: XX / YY（ZZ%）
**判定**: PASS / CONDITIONAL / FAIL
**Critical違反**: なし / [違反項目]

### 指摘事項（CONDITIONAL/FAIL時）
- [ ] [修正が必要な具体的指摘]
- [ ] ...

### 次のアクション
- PASS → Gate X+1 へ進む
- CONDITIONAL → [修正対象] を修正後、再評価
- FAIL → [差し戻し先] から再実行
```

## TDDサイクル全体の品質サマリ

全ゲート通過後、最終サマリを出力する。

```
## TDD Quality Summary

| Gate | Section | スコア | 判定 | 再評価回数 |
|------|---------|--------|------|-----------|
| 1 | コンテキスト分析 | XX% | PASS | 0 |
| 2 | テスト種別判定 | XX% | PASS | 0 |
| 3 | コードレビュー | XX% | PASS | 1 |
| 4 | RED | XX% | PASS | 0 |
| 5 | GREEN | XX% | PASS | 0 |
| 6 | REFACTOR | XX% | PASS | 0 |
| 7 | 目視レビュー | XX% | SKIP | - |
| 8 | 最終検証 | XX% | PASS | 0 |
| 9 | ドキュメント同期 | XX% | PASS | 0 |

**総合スコア**: XX%（全ゲート平均）
**総合判定**: PASS / FAIL
**再評価合計回数**: X回
```
