# TDD Refactor（Gate 6: REFACTOR フェーズ）

> **ロール**: Generator（リファクタ実行・`/decompose`・`/simplify`）→ Evaluator（Gate 6 スコアリング）

---

## チェックリスト

- [ ] 重複コードの除去
- [ ] 命名の改善（変数名、関数名が意図を表現しているか）
- [ ] 関数の分割（1関数1責任）
- [ ] テストコード自体の改善（共通セットアップの抽出など）
- [ ] ファイル肥大化チェック（300行超のファイルがないか）
- [ ] 全テストがパスしたまま

---

## やらないこと（REFACTORフェーズ外）

- 新機能の追加
- パフォーマンス最適化（別タスクとして扱う）
- テストケースの追加（次のREDフェーズで行う）

---

## ファイル構造チェック（`/decompose`）

GREENで追加したコードにより、ソースファイルまたはテストファイルが300行を超えていないか確認する。

**手順:**
1. `wc -l` で変更対象ファイル + テストファイルの行数を確認
2. 300行超のファイルがあれば `/decompose` を呼び出して分割を実施
3. 分割後に全テストを再実行し、パスを確認

**Gate 3（テスト前）との違い:**
- Gate 3: 既存コードの肥大化を**テストを書く前に**解消する
- Gate 6: GREEN実装後に**新たに肥大化した**ファイルを分割する

---

## リファクタ品質チェック（`/simplify`）

REFACTOR完了後、`/simplify` を呼び出してコードの再利用性・効率性・品質を自動チェックする。
指摘があれば修正してからGate 6評価に進む。

---

## Gate 6 評価（Evaluator）

| 評価項目 | 配点 |
|---------|------|
| `[CRITICAL]` 全テストパス維持 | 3 |
| 重複コード除去 | 3 |
| 命名の適切性 | 3 |
| 関数の責務分離 | 3 |
| テストコードの品質 | 3 |
| ファイル構造（300行以下 or `/decompose` 分割済み） | 3 |
| `/simplify` 指摘ゼロ | 3 |
| スコープ逸脱なし | 3 |

**満点24点 / 合格ライン20点（80%）** — Critical項目0点で即FAIL

Gate 6 PASS後:
- E2Eテスト対象がある → `/tdd-ui-review`（UI/UXレビュー）へ
- E2Eなし → Section 6（テスト実行・検証）へ戻る
