# JQIT 商談〜提案ワークフロー

お客様から相談を受けてから、提案書を出すまでの流れ。
はじめて商談を担当する人でも、この手順に沿えば進められる。

### このドキュメントの範囲

```
[本ドキュメント]                [ai-solution-demo]           [jqit-estimate]
 初回の商談〜提案書を出すまで     デモを作って見せる段階         見積書を作る段階
```

初回商談のヒアリングには [ヒアリングシート](hearing-sheet.md) を使う。
デモを作った後のヒアリングには [`ai-solution-demo` のヒアリングガイド](../../ai-solution-demo/references/hearing-guide.md) を使う。

---

## 全体の流れ

```
案件受付 → 事前準備 → 初回商談 → 提案書作成 → 提案提出
 (30分)    (1-2時間)   (60分)    (2-4時間)     (30分)
```

**大事なこと: 初回商談で「形のあるもの」を見せる。**

提案書でもモック画面でもいい。具体物があると話が進む。
事前情報があれば、「たぶんこうだろう」という仮の提案書を作って持っていく。
仮説が外れてもOK。「ここは違います」という反応が得られれば、それ自体が収穫。

## 1. 案件受付（相談が来たら）

相談が来たら、まずこの7つを整理する。全部埋まらなくてOK。

| 確認すること | 例 |
|------------|-----|
| **案件名** | 「〇〇社 - 議事録AI」のように社名+内容 |
| **どんな相談か** | 「議事録作成を自動化したい」くらいでOK |
| **先方の担当者** | 名前・役職・部署 |
| **誰がOKを出すか** | 担当者が決める？上長の承認がいる？ |
| **予算のめど** | 聞けていれば。わからなければ空欄 |
| **いつまでに欲しいか** | 希望時期 |
| **他社にも相談しているか** | 聞けていれば |

### この案件を受けるべきか？

- **進める**: 困っていることが明確 + お金がありそう + 時間に余裕がある
- **まず話を聞く**: 困りごとはぼんやりしているが、お金と判断する人はいる
- **見送りを検討**: お金なし + 期限ギリギリ + 何がしたいか不明確 の3つが揃う場合

## 2. 事前準備（商談前日まで）

### 先方のことを調べる

- 会社のWebサイトで事業内容・規模を確認
- どんなシステムやツールを使っていそうか（採用ページの技術情報が参考になる）
- 過去にうちと取引があったか（Backlog等で確認）

### 「たぶんこうだろう」を考える（仮説づくり）

お客様の状況を想像して、「今こうなっていて → こうなりたいはず」を考える。

```
【今の状態（予想）】
- 〇〇の作業を手でやっている
- △△のデータが個人のPCに散らばっている

【こうなったら嬉しいはず】
- AIで〇〇を自動化して、△△時間を減らす
- データを一箇所にまとめて、すぐ見られるようにする
```

完璧じゃなくていい。「こういうことでお困りですか？」と聞ける材料があればOK。

### 提案書を先に作る（できれば）

```
# Claude Code で生成
/jqit-proposal
```

仮説をもとに、ざっくり版の提案書を用意しておく。

### デモ・モック画面の準備（余裕があれば）

| 準備するもの | かかる時間 | 使うツール |
|------------|----------|----------|
| 提案書（PPTX） | 30分〜1時間 | `/jqit-proposal` |
| 画面イメージ（動かないHTML） | 1〜2時間 | Claude Code |
| 動くデモ | 2〜4時間 | Claude Code + Vercel |

先方が本気（お金がある・決める人が出てくる）なら、デモまで作る価値あり。

## 3. 初回商談（60分）

### 時間配分の目安

| 時間 | やること |
|------|---------|
| 最初の5分 | 自己紹介、JQITの簡単な紹介 |
| 5〜20分 | **ヒアリング**（お客様の話を聞く） |
| 20〜40分 | **提案・デモ**（用意したものを見せる） |
| 40〜50分 | **すり合わせ**（反応を聞いて方向性を確認） |
| 最後の10分 | **次のアクション**（次回日程・宿題の確認） |

### ヒアリング（お客様の話を聞く）

[ヒアリングシート](hearing-sheet.md) を手元に置いて進める。最低限これだけは聞く:

1. **何に困っているか**
2. **なぜ今やりたいのか**（きっかけは何か）
3. **誰がOKを出すか**
4. **いつまでに必要か**
5. **予算はあるか**（ざっくりでOK）

**聞き方のコツ**:
- 「具体的には？」「例えば？」で掘り下げる
- 数字で聞く（「月に何件？」「何時間かかる？」）
- お客様の言葉をそのまま使って確認する（「つまり〇〇ということですか？」）

### 提案・デモを見せる

事前に作った提案書やデモを見せながら、こんな流れで話す:

1. **お客様の課題を整理**（「こういうお困りごとがありますよね」）
2. **解決のイメージを提示**（「こんな感じで解決できます」）
3. **ざっくりのスケジュール感**（「まず2ヶ月でここまで → その後拡張」）
4. **費用のめど**（「だいたい〇〇万円〜の規模感です」※正式見積は後日）

お客様のメリット（時間が減る・コストが下がる・ミスが減る）を中心に話す。
技術の話に偏りすぎない。

### 商談の終わり方（一番大事）

**必ず「次の約束」をして終わる。**

- [ ] 次回の打ち合わせ日を仮押さえした
- [ ] JQIT側の宿題を確認した（提案書・見積もり・デモ等）
- [ ] 先方側の宿題を確認した（資料提供・社内確認等）
- [ ] それぞれの期限を決めた

## 4. 提案書をつくる（商談後）

### まず商談メモを書く（当日中に）

```markdown
# 商談メモ: 〇〇社 — 案件名
日付: YYYY-MM-DD

## お客様の課題（今の状態）
-

## こうなりたい（理想の状態）
-

## 条件
- 予算:
- 希望時期:
- 決める人:
- 他社にも相談している？:

## 提案やデモへの反応
- 良かった点:
- 違った点:
- 追加の要望:

## 次のアクション
- [ ] JQIT: （期限: ）
- [ ] 先方: （期限: ）
- [ ] 次回打ち合わせ: 月/日
```

### 正式な提案書を作る

```
# Claude Code で生成
「以下の商談メモをベースに正式提案書を作ってください」
+ 商談メモ.md を添付

/jqit-proposal
```

**提案書の標準構成（8枚）**:

| スライド | 内容 |
|---------|------|
| 表紙 | 案件名、日付 |
| 課題の整理 | お客様の困りごと |
| 解決策 | 提案するAIソリューション |
| システムの全体像 | どう動くかの図 |
| 主な機能 | できることの一覧 |
| スケジュール | いつ何をやるか |
| 費用 | ざっくりの金額 |
| 次のステップ | この後どう進めるか |

### やること・やらないことを整理する（スコープ定義書）

提案書とは別に、「ここまでやります / ここはやりません」を書いた資料を作る。
後から「これも含まれてると思った」というトラブルを防ぐため。

```markdown
# やること・やらないこと: 〇〇案件

## やること
- [ ] 機能A: ...
- [ ] 機能B: ...

## やらないこと（別途相談）
- スマホアプリ対応
- 多言語対応
- 大量データの移行

## 前提条件（お客様にお願いすること）
- 〇〇のデータを提供いただくこと
- テスト用の環境をご用意いただくこと
```

## 5. 提案書を出す

### 出す前のチェック

- [ ] 提案書の全ページを開いて表示が崩れていないか確認した
- [ ] やること・やらないことの資料を添付した
- [ ] 見積書を添付した（`/jqit-estimate` で作成）
- [ ] 送付メールの文面を確認した

### 送るときの注意

- 商談で約束した期日を守る
- PPTX + PDF の両方を送ると親切
- **次回打ち合わせの日程を改めて確認**する

## よくあるケースと対応

### 「とりあえず話を聞きたい」と言われた場合
→ ヒアリング中心の商談にする。提案は次回。
→ ただし「AIでこんなことができます」の事例スライド（3〜5枚）は持っていく。

### 要件書（やりたいことの資料）が先にもらえた場合
→ 提案書 + 見積書を初回商談で出せるチャンス。先に作って持っていく。

### 「予算がない」「まだ決まっていない」と言われた場合
→ 小さく始める提案をする（「まず〇〇万円でここまで」）。
→ 「これだけコストが下がります」という数字を出して、予算確保の材料にしてもらう。

### 「急いでいる」と言われた場合
→ やることを絞って、最速で動くものを出す提案にする。
→ 「2週間でデモ → 2ヶ月で本番」のように段階的に。
