# ヒアリングガイド

顧客先でのデモ提示・提案時に確認すべき項目のマスターリスト。

## 使い方

- **営業**: カテゴリ1, 2, 3, 5を中心に確認
- **開発**: カテゴリ2, 3, 4, 5を中心に確認
- 各質問の【対象】【タイミング】【参照成果物】を参考に、状況に応じて取捨選択する

### タイミングの目安

| タイミング | 状況 | 参照可能な成果物 |
|-----------|------|----------------|
| **デモ提示時**（Phase 5-7完了後） | 動くプロトタイプ + デモ動画を見せながら | requirement.md, design/*.md, wireframes/, demo/*.mp4 |
| **提案書提出時**（Phase 9完了後） | 正式提案の場で | 上記 + proposal/*.pptx, manual.pdf |

---

## カテゴリ1: ビジネス要件

### 予算・コスト
> 【対象】営業 【タイミング】両方 【参照成果物】proposal/*.pptx

- [ ] 予算枠は確保済みか、これから申請か
- [ ] 概算でどの程度の投資を想定しているか（初期費用・ランニング）
- [ ] ROIの評価基準は何か（コスト削減額、業務時間短縮、売上向上 等）
- [ ] 予算の承認時期はいつ頃か

### 意思決定プロセス
> 【対象】営業 【タイミング】両方 【参照成果物】なし

- [ ] 最終決裁者は誰か（役職・部署）
- [ ] 承認フロー（稟議が必要か、何段階あるか）
- [ ] 意思決定の期限・目標時期
- [ ] 比較検討中の他社・他案はあるか
- [ ] 導入判断のキラー要因は何か（コスト、機能、実績 等）

### スケジュール
> 【対象】営業 【タイミング】両方 【参照成果物】なし

- [ ] 希望する本番稼働時期
- [ ] 社内イベント・期末等のマイルストーンがあるか
- [ ] 段階的導入（パイロット→全社展開）を想定しているか
- [ ] パイロット対象の部署・人数

### 競合・既存システム
> 【対象】営業 【タイミング】デモ提示時 【参照成果物】なし

- [ ] 現在同種の業務をどう行っているか（手作業・既存ツール）
- [ ] 既存ツールへの不満点
- [ ] 他社からの提案を受けている場合、その評価ポイント

## カテゴリ2: 機能フィードバック

### デモへの反応
> 【対象】両方 【タイミング】デモ提示時 【参照成果物】requirement.md の機能一覧, demo/*.mp4

- [ ] デモを見て想定通りだった点はどこか
- [ ] 想定と違った点・期待外れだった点はどこか
- [ ] 最も価値を感じた機能はどれか
- [ ] 「これは使わない」と感じた機能はあるか

### 要件の再確認
> 【対象】両方 【タイミング】デモ提示時 【参照成果物】requirement.md（F-xxx一覧）

- [ ] 各要件（F-xxx）の優先度は変更ないか（Must/Should/Could）
- [ ] デモで確認できなかった要件について、認識は合っているか
- [ ] 新たに追加したい機能はあるか
- [ ] 削除・縮小してよい機能はあるか

### データ・コンテンツ
> 【対象】両方 【タイミング】デモ提示時 【参照成果物】design/db-schema.md

- [ ] デモのサンプルデータは実際の業務データに近いか
- [ ] 実際に扱うデータ量の規模感（レコード数、ファイル数）
- [ ] データの入力方法（手入力、CSV取込、API連携 等）

## カテゴリ3: UX/デザイン

### 画面の使い勝手
> 【対象】両方 【タイミング】デモ提示時 【参照成果物】wireframes/, design/screen-design.md

- [ ] 画面レイアウトは直感的か、迷う箇所はないか
- [ ] 文字サイズ・配色は見やすいか
- [ ] 操作ステップ数は適切か（多すぎ・少なすぎ）
- [ ] 頻繁に使う機能へのアクセスは容易か

### ブランド・デザイン適合
> 【対象】営業 【タイミング】デモ提示時 【参照成果物】design/design-system.md

- [ ] 自社のブランドガイドライン（ロゴ、カラー、フォント）はあるか
- [ ] デモの配色・デザインテイストは許容範囲か
- [ ] 社内規定でUIに関する制約はあるか

### デバイス・アクセシビリティ
> 【対象】両方 【タイミング】デモ提示時 【参照成果物】requirement.md の非機能要件

- [ ] 主な利用デバイスは何か（PC、タブレット、スマートフォン）
- [ ] モバイル対応の必要度（必須 / あれば嬉しい / 不要）
- [ ] アクセシビリティ要件はあるか（WCAG準拠、多言語対応 等）

## カテゴリ4: 技術要件

### 既存システム連携
> 【対象】開発 【タイミング】両方 【参照成果物】design/api-design.md

- [ ] 連携が必要な既存システム一覧（基幹、CRM、メール 等）
- [ ] 各システムのAPI公開状況（REST/SOAP/ファイル連携）
- [ ] 認証方式（OAuth2、APIキー、SAML/SSO 等）
- [ ] データ連携の頻度（リアルタイム、バッチ、手動）

### セキュリティ
> 【対象】開発 【タイミング】提案書提出時 【参照成果物】requirement.md の非機能要件

- [ ] セキュリティポリシー・チェックリストの有無
- [ ] データの保管場所の制約（国内DC限定、オンプレ必須 等）
- [ ] 個人情報・機密情報の取り扱い範囲
- [ ] セキュリティ監査・第三者認証の要否
- [ ] IP制限、VPN接続等のネットワーク要件

### インフラ・パフォーマンス
> 【対象】開発 【タイミング】提案書提出時 【参照成果物】design/db-schema.md, design/api-design.md

- [ ] 想定同時接続ユーザー数
- [ ] レスポンスタイムの期待値
- [ ] 可用性の要求水準（99.9%等のSLA）
- [ ] クラウド利用のポリシー（AWS/GCP/Azure指定、マルチクラウド 等）
- [ ] 既存のインフラ環境・制約

## カテゴリ5: 運用・保守

### サポート体制
> 【対象】両方 【タイミング】提案書提出時 【参照成果物】proposal/*.pptx

- [ ] 期待するサポート形態（メール、チャット、電話、オンサイト）
- [ ] 対応時間帯の希望（平日営業時間、24/365 等）
- [ ] SLA（障害対応の応答時間・復旧時間）の希望
- [ ] エスカレーション先の想定

### トレーニング・導入支援
> 【対象】両方 【タイミング】提案書提出時 【参照成果物】docs/manual.pdf

- [ ] エンドユーザー向けトレーニングの必要性
- [ ] 管理者向け研修の必要性
- [ ] マニュアル・ヘルプの提供形式の希望
- [ ] 導入支援の期間と範囲

### データ移行
> 【対象】開発 【タイミング】提案書提出時 【参照成果物】design/db-schema.md

- [ ] 既存データの移行が必要か

- [ ] データクレンジングの必要性
- [ ] 移行時のダウンタイム許容範囲

### 継続的改善
> 【対象】両方 【タイミング】提案書提出時 【参照成果物】なし

- [ ] 本番リリース後の機能追加の想定
- [ ] フィードバック収集の仕組みへの関心
- [ ] 将来的なスケールアップの想定（ユーザー数、データ量）
- [ ] 契約形態の希望（スポット / 保守契約 / 内製化支援）
