# Consistency Review Checklist

Phase 3（整合性レビュー）で使用するチェックリスト。5カテゴリで照合を行う。

## Category 1: 網羅性（memo → requirement）

memo.mdの全項目がrequirement.mdでカバーされているか。

| チェック項目 | 判定基準 |
|-------------|---------|
| ヒアリング項目の網羅 | memo.mdの各段落/箇条書きに対応する要件(F-xx)が存在する |
| 暗黙の要件抽出 | memo.mdの文脈から読み取れる暗黙要件が明示化されている |
| スコープ外の明記 | 対象外と判断した項目がスコープ外として記載されている |
| 優先度の妥当性 | Must/Should/Couldの振り分けがmemo.mdの温度感と整合している |

**レビュー手順**:
1. memo.mdをセンテンス単位で分割
2. 各センテンスに対応するF-xxを特定
3. 対応なしのセンテンスを「未カバー項目」として報告
4. カバレッジ率（カバー済み/全項目）を算出

---

## Category 2: 網羅性（requirement → design）

requirement.mdの全要件がdesign/*.mdで設計されているか。

| チェック項目 | 判定基準 |
|-------------|---------|
| 要件→設計の対応 | 各F-xxに対応するS-xxが存在する |
| 画面設計の網羅 | 要件で言及される画面が screen-design.md に存在する |
| DB設計の網羅 | 要件で言及されるデータが db-schema.md のテーブルに存在する |
| API設計の網羅 | 要件で言及される操作が api-design.md のエンドポイントに存在する |

**レビュー手順**:
1. requirement.mdの全F-xxをリスト化
2. design/*.mdの全S-xxとの対応表を作成
3. 対応なしのF-xxを「設計漏れ」として報告
4. カバレッジ率を算出

---

## Category 3: トレーサビリティマトリクス

F-xx（要件）→ S-xx（設計）→ A-xx（UI/アクション）の追跡可能性。

| チェック項目 | 判定基準 |
|-------------|---------|
| F→S対応 | 全F-xxにS-xxが紐付いている |
| S→A対応 | 全S-xxにA-xxが紐付いている（Phase 4以降） |
| 逆追跡 | S-xxから元のF-xxを辿れる |
| 孤立設計の検出 | F-xxに紐付かない孤立したS-xxがない |

**出力フォーマット**:
```markdown
| 要件ID | 要件名 | 設計ID | アクションID | 状態 |
|--------|--------|--------|-------------|------|
| F-001  | ログイン機能 | S-001, S-002 | A-001 | OK |
| F-002  | データ入力 | S-003 | - | 設計のみ |
| F-003  | レポート出力 | - | - | 未設計 |
```

---

## Category 4: 数値・用語の一貫性

ドキュメント間で数値や用語が一致しているか。

| チェック項目 | 判定基準 |
|-------------|---------|
| 予算・金額 | memo.md記載の予算とrequirement.mdの記載が一致 |
| 人数・規模 | ユーザー数・データ量の記載が一致 |
| 期限・スケジュール | 納期や目標日付の記載が一致 |
| 用語の統一 | 同じ概念に異なる用語が使われていない |
| 固有名詞 | 会社名・システム名・部署名が正確に統一 |

**レビュー手順**:
1. 数値パターン（金額、人数、日付）を各ドキュメントから抽出
2. ドキュメント間で照合
3. 不一致を「数値不一致」として報告
4. 用語リストを生成し、揺れを検出

---

## Category 5: 矛盾検出

ドキュメント間の論理的矛盾を検出する。

| チェック項目 | 判定基準 |
|-------------|---------|
| 機能の矛盾 | requirement.mdで「必須」の機能がdesign.mdで「対象外」になっていない |
| 技術制約の矛盾 | 技術選定が要件の制約と矛盾していない |
| 非機能要件の矛盾 | パフォーマンス要件とDB設計/API設計が矛盾していない |
| フロー上の矛盾 | 画面遷移と業務フローが論理的に矛盾していない |

**レビュー手順**:
1. 各ドキュメントの主張・前提を箇条書きで抽出
2. 主張間の論理的整合性を検証
3. 矛盾を「重大度: High/Medium/Low」で分類
4. 矛盾の内容と修正提案をレポートに記載

---

## Review Report Format

レビュー結果は以下のフォーマットで `docs/reviews/review-report.md` に出力する:

```markdown
# 整合性レビューレポート

**実行日時**: YYYY-MM-DD HH:MM
**対象フェーズ**: Phase 1-2
**判定**: PASS / FAIL / CONDITIONAL PASS

## サマリー

| カテゴリ | 結果 | 重大問題 | 軽微問題 |
|---------|------|---------|---------|
| 1. 網羅性(memo→req) | PASS/FAIL | 0 | 0 |
| 2. 網羅性(req→design) | PASS/FAIL | 0 | 0 |
| 3. トレーサビリティ | PASS/FAIL | 0 | 0 |
| 4. 数値・用語一貫性 | PASS/FAIL | 0 | 0 |
| 5. 矛盾検出 | PASS/FAIL | 0 | 0 |

## 詳細

### [カテゴリ名]

#### [問題ID] [重大度]
- **内容**: ...
- **該当箇所**: ...
- **修正提案**: ...

## 推奨アクション

1. ...
```
